Prompt 工程最容易被误解成“写一句更聪明的话”。

很多人第一次接触大模型时，都会经历一个阶段：到处收集提示词模板，研究各种框架缩写，试图找到一条万能咒语。好像只要句式足够完整，模型就会稳定、准确、听话。

但我现在更愿意把 Prompt 工程理解成另一件事：**设计模型的工作环境**。

一个好的 Prompt，不只是告诉模型“回答我”。它是在给模型布置一个临时工作间：你是谁，你要解决什么问题，你能看到哪些材料，哪些动作可以做，哪些事情不能做，输出要长成什么样，以及结果会如何被检查。

这也是为什么同一个模型，换一个 Prompt，表现会像换了一个人。

## 模型不缺能力，缺的是任务现场

如果只说“帮我修一下这个登录接口 500”，模型当然能回答。

问题是，它不知道这是前端报错、后端异常、网关问题，还是数据库连接失败。它也不知道你希望它给排查思路、直接改代码、解释日志，还是写一份事故复盘。

在这种情况下，模型最容易做的事就是补全一个“平均意义上的 bug 分析”。它会猜测鉴权、跨域、参数校验、数据库、token 过期，每一条都像那么回事，但不一定和你的现场有关。

这不是因为模型不会排障，而是因为你没有给它一个足够明确的任务现场。

当 Prompt 里出现技术栈、错误日志、相关代码、最近改动、排查目标和输出格式后，事情就变了。模型不再是在“猜一个接口为什么会挂”，而是在一个被约束过的环境里工作：

```markdown
# 角色

你是一名资深 TypeScript 后端工程师，擅长定位线上接口异常。

# 任务

请根据下面的信息，分析登录接口返回 500 的最可能原因，并给出最小修复建议。

# 现场信息

- 技术栈：NestJS + Prisma + PostgreSQL
- 接口：POST /api/login
- 现象：请求返回 500，但用户名和密码参数都存在
- 错误日志：Cannot read properties of undefined (reading 'compare')
- 最近改动：把 PasswordService 从 AuthModule 中拆到了 SharedModule
- 相关范围：只分析依赖注入、模块导出、登录流程，不要展开数据库性能问题

# 输出要求

1. 先给出最可能原因
2. 再给出 3 个验证步骤
3. 最后给出最小修复方向
4. 不要编造未提供的文件名或代码
```

这段 Prompt 的关键不在于它用了 Markdown，也不在于它看起来“高级”。关键在于它减少了模型需要猜的东西。

Prompt 工程的第一条原则不是“写详细”，而是：**把模型必须知道、但无法自己可靠推断的信息放进来**。

## 结构化不是为了好看，而是为了减少误解

结构化 Prompt 的价值，经常被说得太形式化。

RTF、ICIO、CO-STAR、CRISPE、TIDD-EC、BROKE，这些框架当然有用。但如果只是背缩写，很快就会变成新的形式主义。

这些框架真正解决的是同一个问题：**让模型知道每段信息的用途**。

角色不是装饰。它决定模型应该调用哪一类语言风格和领域知识。

任务不是标题。它决定模型到底要生成、分析、分类、提取，还是决策。

上下文不是材料堆。它决定模型应该基于什么信息回答，而不是自由发挥。

输出格式不是排版要求。它决定结果能不能被人快速阅读，或者被程序继续处理。

我觉得一个更简单的判断方式是：如果你的 Prompt 里有一段信息，但模型不知道这段信息该如何使用，那它就只是噪声。

结构化的目的，是给信息贴上标签。

```markdown
# 背景

这是一篇写给技术团队内部阅读的复盘文章。

# 目标

解释为什么这次故障暴露的是流程问题，而不是单点代码问题。

# 边界

不要追责个人，不展开无关的组织问题。

# 输出

写成 1200 字以内的中文博客，语气克制，结论明确。
```

这类 Prompt 不复杂，但很稳定。

它没有试图把模型“调教”成一个完美作者，而是把写作环境收窄到一个具体任务里。

## 示例不是答案，而是行为锚点

Few-shot 的价值也经常被低估。

很多人提供示例，是为了让模型“照着写”。但更准确地说，示例是在告诉模型：**什么样的输入应该对应什么样的输出行为**。

比如让模型写“苹果风格”的矿泉水广告文案，如果只说“模仿苹果风格”，模型很可能写出一些泛泛的高级感词汇。因为“苹果风格”这个词本身太模糊了。

但如果给它几条短句示例：

```text
轻到无感。强到出众。
性能强劲。设计轻盈。
突破想象。日常不同。
```

模型就会更容易抓到节奏：短句、克制、对称、留白。

示例的作用不是提供素材，而是提供模式。

这在分类、路由、工具调用里更明显。简单任务可以 zero-shot，因为模型知道常识边界。复杂任务如果没有示例，模型很难知道“查询 API”“检索知识库”“联网查询”之间的边界在哪里。

少量好的示例，往往比一大段抽象规则更有效。

不过示例也有代价。

它会占用上下文，会让 Prompt 变长，也会引入位置敏感性。不同模型对示例放在系统提示词开头、用户消息开头、用户消息末尾的反应可能不同。小模型尤其明显。

所以示例不是越多越好。它应该只在任务边界容易混淆、输出风格需要稳定、或者模型经常走错路时出现。

## 复杂技巧的本质是把不稳定变成流程

CoT、Self-Consistency、Tree of Thoughts、Reflection、ReAct 这些方法，表面上是提示词技巧，往深一点看，其实是在把模型的不稳定行为流程化。

CoT 让模型先分解问题，再给结论。它的价值不只是提高答对率，更重要的是暴露推理路径。只要路径可见，人就能判断模型错在理解、分解、计算，还是结论。

Self-Consistency 不是让模型更聪明，而是用多次推理抵消单次偶然错误。它的逻辑很工程化：同一个问题跑多次，取一致性更高的答案。

Tree of Thoughts 适合方案设计，因为复杂问题往往不止一条路径。它让模型先生成多个候选方案，再比较优缺点，而不是沿着第一个想法一路走到底。

Reflection 则更像一次内部评审：先写初稿，再审查，再修订。它适合高质量内容生成、方案复盘、代码审查这类任务，但不适合对延迟敏感的实时场景。

ReAct 更进一步。它把“想”和“做”连接起来：先推理下一步，再调用工具，再观察结果，再继续调整。到了这里，Prompt 已经不只是输入文本，而是智能体系统的一部分。

这些方法真正要解决的不是“怎么让模型说得更像人”，而是：

**当任务复杂到一次回答不可靠时，如何把回答过程拆成可观察、可修正、可验证的步骤。**

代价也很明确。

多轮调用会变慢，成本会上升，上下文会膨胀，流程设计也会变复杂。一个简单摘要任务不需要 Reflection，一个普通分类任务也不需要 Tree of Thoughts。

复杂技巧应该用在复杂性真的存在的地方。

## 评测让 Prompt 从手感变成资产

Prompt 调优很像“炼丹”，但不能永远停留在手感。

手感有价值。尤其在内容质量、逻辑表达、语气是否自然这些事情上，人类判断仍然比很多自动指标更直接。

但如果一个 Prompt 要进入真实产品，光靠“这次看起来不错”是不够的。你需要知道它在多数场景下是否稳定。

人工评测最适合回答这些问题：

- 输出是否相关。
- 信息是否准确。
- 逻辑是否连贯。
- 内容是否完整。
- 是否存在有害或不安全表达。
- 风格是否符合预期。

自动评测更适合边界清晰的任务。比如分类任务可以看准确率、召回率、F1；检索任务可以看 Precision@K、Recall@K；摘要和翻译可以参考 ROUGE、BLEU、BERTScore。

现在更实用的一类方法，是用更强模型做裁判。

你可以让一个更稳定的模型根据评分准则判断两个输出哪个更好，或者给某个输出按相关性、准确性、完整性打分。这不能完全替代人工，但它能让 Prompt 迭代从“凭感觉改”变成“带标准地改”。

Prompt 一旦能被评测，就不再只是一次性的文本。

它会变成可以版本管理、可以回归测试、可以持续优化的工程资产。

## nof1.ai 的价值不在交易，而在 Prompt 的边界设计

最近比较有意思的案例是 [nof1.ai](https://nof1.ai/)。

它给不同大模型一笔初始资金，让模型在加密货币市场中做交易决策。这里不讨论交易收益，也不建议任何人直接让模型操作真实资金。这个案例真正值得看的地方，是它展示了一个复杂 Prompt 如何把模型约束进一个高风险任务环境里。

这个系统提示词很长，但核心结构并不神秘。

它先定义模型身份：你是一个自主加密货币交易代理。然后定义交易环境：交易所、资产范围、初始资金、决策频率、杠杆范围。接着定义动作空间：只能开多、开空、持有、平仓。再定义仓位管理、风险管理、输出 JSON 格式、技术指标解释、模型不能访问的信息、常见错误、交易原则。

这不是在写一段“聪明的提示词”。

这是在写一份操作手册。

它的强度来自边界：

- 模型只能在四个动作里选择。
- 每次交易必须给出止盈、止损、失效条件、信心分数和风险金额。
- 输出必须是固定 JSON，方便下游系统解析。
- 所有时间序列数据都明确说明从旧到新排列，避免模型读反。
- 系统明确告诉模型没有新闻、没有社交媒体、没有历史记忆，不能假装知道。
- Sharpe Ratio 被作为反馈信号，帮助模型校准风险偏好。

这个案例几乎把 Prompt 工程的核心都压在一起了：角色、任务、上下文、动作边界、输出格式、负向约束、领域知识、反馈指标。

真正重要的是，它没有指望模型“自己懂交易纪律”。

它把纪律写进了环境。

## 好 Prompt 的目标不是让模型自由发挥

很多 Prompt 写不好，是因为人把模型当成一个“什么都懂的聪明人”，然后期待它自己补齐所有隐含前提。

但在工程里，更可靠的做法往往相反。

不要让模型猜任务。

不要让模型猜边界。

不要让模型猜输出格式。

不要让模型猜什么信息可信，什么信息只是网页、日志、历史对话里的噪声。

好的 Prompt 不是让模型自由发挥，而是让模型在正确的自由度里发挥。

如果任务很简单，就给清晰指令。

如果任务边界容易混淆，就给示例。

如果输出要进入程序，就给严格格式。

如果任务需要专业判断，就给上下文和角色。

如果任务风险高，就给负向约束、验证机制和人工确认。

如果一次回答不可靠，就把它拆成推理、行动、观察、反思和评测。

## 我现在对 Prompt 工程的几个判断

Prompt 工程不是模板工程。模板只是把一些好习惯固定下来。

Prompt 工程也不是语言艺术。写得优雅不重要，模型能不能稳定理解和执行才重要。

Prompt 工程更像接口设计。你在定义人、模型、工具和数据之间如何交接信息。

它也像工作流设计。你在决定哪些事情一次完成，哪些事情要分步骤，哪些地方要评审，哪些地方要回退。

它最终还是一件工程工作：观察失败，缩小歧义，补充上下文，约束输出，建立评测，然后继续迭代。

所以我不太相信所谓万能 Prompt。

我更相信具体任务里的具体环境设计。

当一个 Prompt 能清楚说明任务、边界、材料、格式和反馈方式时，模型就不再只是“回答问题”。

它开始像是在工作。
