Function Calling 最容易被误解成一句话：让大模型去调用函数。

但真正做过之后会发现，模型并没有“执行”任何函数。它不会真的去查天气、退订单、访问数据库，也不会替你的系统完成鉴权、幂等、异常处理。它做的事情更窄，也更关键：根据用户意图，判断该用哪个工具，并生成一组结构化参数。

真正的函数执行，仍然发生在开发者自己的应用里。

所以，Function Calling 的本质不是“模型长出了手”，而是开发者给模型设计了一套可理解、可验证、可执行的协作协议。

这个理解非常重要。因为一旦把它当成“模型自己会干活”，系统就会变得不可控；而把它当成协议设计，工程边界就清楚了很多。

## 模型负责判断，应用负责执行

大模型自身有几个天然限制：它不知道实时天气，不知道你的私域订单，不知道数据库里的库存状态，也不能直接操作你公司的退款系统。

Function Calling 正是为了解决这个问题。

一个典型流程是这样的：

1. 应用把用户问题、历史消息、可用工具定义一起发给模型。
2. 模型返回一个 tool call，里面包含工具名和参数。
3. 应用端根据工具名找到真实函数，并执行代码。
4. 应用把函数返回结果再发回模型。
5. 模型基于结果生成最终回复，或者继续请求下一次工具调用。

这里最容易忽略的是第三步：函数不是模型执行的，是你的代码执行的。

模型输出的是一种“调用意图”。比如它说要调用 `get_weather`，参数是 `location = Paris`。至于这个函数是否存在、用户是否有权限、参数是否合法、接口是否超时、结果如何兜底，全部是应用层要处理的事情。

换句话说，Function Calling 并没有降低工程责任。它只是把自然语言理解和结构化决策这部分交给了模型。

## 工具定义，就是给模型看的接口文档

以前我们写接口，很多时候注释可以少一点。因为调用方也是人，或者就是我们自己的代码。函数名不够清晰，参数意义不够完整，最多是维护时多看几眼上下文。

但给模型用的工具不是这样。

模型不会读你的业务脑图，也不知道你们内部约定。它只能通过工具名、描述、参数 schema、参数说明来判断这个工具该不该用、怎么用。

所以一个好的工具定义，本质上是一份给模型看的接口文档。

工具名要直接。`apply_refund` 比 `handle_order` 清楚得多。描述要说明什么时候用，而不只是说它做什么。参数描述要告诉模型格式、含义、边界，而不是只写一个字段名。

参数结构也要克制。

尽量扁平，少嵌套。JSON 层级越深，模型越容易填错。参数数量也不要太多，最好控制在三四个以内。默认值、鉴权信息、上下文里已经有的信息，不应该让模型传。那些事情应该由服务端补全。

还有一个很实用的技巧是枚举。

如果某个参数只有有限选项，就把它从“问答题”变成“选择题”。比如餐厅搜索里的 `good_for` 可以限定为 `families`、`business_meeting`、`solo_dining` 等。这样模型更容易把“适合带孩子”映射成 `families`，而不是临场编一个新值。

模型最怕的不是不会填参数，而是以为自己会填。

枚举、严格 schema、few-shot，本质上都是在减少模型自由发挥的空间。

## 好工具要能失败得明白

很多工具调用失败，并不是模型完全错了，而是它拿到的反馈太差。

如果工具内部只是抛一个异常，或者返回一句“参数错误”，模型很难修正。它不知道是订单号格式错了、状态不允许退款、商品不在售后期，还是用户权限不足。

面向模型的错误提示，应该像面向一个会读文字但不了解系统内部的人。

比如“订单号格式错误，应为纯数字字符串”，就比“invalid param”有用得多。再比如“该订单已经发起过退款，请不要重复申请”，就能帮助模型在下一轮回复中安抚用户，而不是继续尝试调用同一个工具。

返回值也是一样。

不要把复杂对象原封不动丢给模型，也不要返回只有系统才能理解的状态码。工具返回值应该清晰表达结果：做了什么、是否成功、下一步是什么、用户是否还需要操作。

Function Calling 的稳定性，很大程度上取决于工具结果能不能被模型读懂。

## 电商退款案例：难点不在退款，而在判断何时退款

以“自动帮买家申请退款”为例，真正的难点不是调用退款接口。

如果系统里已经有 `refund(orderId, reason)` 服务，把它包装成一个 `apply_refund` 工具并不复杂。难的是：什么时候应该调用它？

用户说“质量太差了”，可能是严重质量问题；用户说“不喜欢”，可能只是主观偏好；用户说“尺寸不合适”，也要区分是自己买错了，还是商品描述不符。

这时候 Prompt 就不是简单的角色扮演，而是在定义业务流程。

一个更可靠的客服 Prompt 至少要包含几类约束：

角色上，它是电商平台客户体验专家，不是闲聊助手。

任务上，它要先识别用户是否表达了严重商品质量不满，比如“根本没法用”“是坏的”“有瑕疵”“一用就坏了”。

流程上，它不能一上来就退款，而是先用封闭式问题确认具体质量问题。比如：“您是说刚收到的衣服袖口已经完全开线了，对吗？”

触发条件上，只有当用户确认了具体质量问题，才调用退款工具。

限制上，对于“不喜欢”“物流慢”“普通尺寸不合适”这类问题，不应该直接退款，而是走常规客诉流程。

这就体现了 Function Calling 的价值：模型不是单纯把一句话转成接口参数，而是在对话中完成判断、确认、执行、安抚的协作。

下面这个工具定义看起来很简单，但它已经把动作边界收窄到了“申请退款”这一件事：

```java
@Component
public class OrderTools {

    @Autowired
    private OrderManageService orderManageService;

    @Tool(name = "apply_refund", description = "根据用户传入的订单信息发起退款")
    public String refund(
            @ToolParam(description = "订单编号，为数字类型") String orderId,
            @ToolParam(description = "商品名称") String name,
            @ToolParam(description = "退款原因") String reason) {

        // 真实退款动作仍然由应用服务执行，模型只负责生成调用意图和必要参数。
        orderManageService.refund(orderId, reason);

        // 返回值要让模型能直接读懂，便于后续向用户解释处理结果。
        return "已为商品：" + name + "，订单号：" + orderId + " 申请退款，退款原因：" + reason;
    }
}
```

真正的安全性，不在这个注解本身，而在服务端内部继续完成权限校验、订单状态校验、幂等控制和日志记录。

## 对话记忆让工具调用有上下文

真实客服场景一定是多轮的。

用户第一轮进入售后时，系统可能已经知道用户 ID、订单号、对话 ID、当前状态。后面用户只会说“这个坏了”“就是刚才那个”“对的”，不会每句话都重新报订单号。

如果没有对话记忆，模型就会丢失上下文。

在 Spring AI 里，可以用 `chatId` 作为 `CONVERSATION_ID`，把订单号、用户 ID、对话状态放进第一轮上下文。后续每次对话都带上同一个 `chatId`，模型就能在需要调用退款工具时，把记忆里的订单信息一起用上。

这一步看起来像基础设施，但它决定了工具调用是否能落到真实业务对象上。

没有记忆，Function Calling 很容易变成一次性的参数提取；有了记忆，它才更像一个连续的业务流程。

## 结构化输出是给前端和系统看的

Function Calling 不只是后端和模型之间的事。

比如用户第一次进入售后对话时，后端可能需要返回固定结构：订单号、用户 ID、对话 ID、对话状态。前端拿到这些字段之后，才能进入正确页面、维护会话状态、展示对应 UI。

这就是结构化输出的价值。

自然语言适合给用户读，结构化输出适合给系统读。两者不能混在一起。

如果模型返回一段看起来很自然的话：“好的，我已经为您创建了对话”，前端其实很难稳定解析。但如果返回固定对象，系统就能继续往下走。

所以在工程里，结构化输出和 Function Calling 往往是配套出现的：前者让模型输出可解析，后者让模型意图可执行。

## 流式输出和工具调用放在一起，会暴露模型差异

实战里还有一个很容易踩的坑：流式输出。

不同模型返回 tool call 的方式不完全一样。有些模型会一次性返回完整函数名和参数；有些模型会把工具名、arguments 拆成多个 chunk 分段返回。

如果框架处理得不好，就可能出现很怪的问题：一次返回里有工具名但没参数，另一次有参数但没工具名，最后工具调用失败。

这不是业务逻辑错了，而是流式 tool call 的拼接逻辑没有处理好。

正确做法应该是：检测到 tool call 后，持续汇聚分段返回的 arguments，直到工具调用信息完整，再执行真实函数。

这也说明，Function Calling 的可靠性不只取决于 Prompt 和工具定义，还取决于模型实现、框架适配、流式协议处理这些更底层的细节。

同样的业务代码，换一个模型就能工作，未必是玄学。很多时候只是模型对 tool call chunk 的返回方式不同，而框架没有兼容好。

## 不要把复杂工具链交给模型硬串

Function Calling 很容易让人兴奋，然后开始设计一长串工具：

先查订单，再查商品，再查物流，再判断售后政策，再申请退款，再发通知，再改状态。

这在图上很好看，在真实对话里会变脆。

多轮对话中，工具结果不会自动变成下一轮的可靠上下文。你需要自己管理结果、传递状态、处理失败。链路越长，模型越容易在中间丢信息、误用结果，或者在不该调用的时候继续调用。

更稳的做法是：把强相关的步骤封装到一个更高层的工具里。

比如不要让模型分别调用“查订单”“校验售后”“申请退款”“记录日志”。可以提供一个 `apply_refund`，由服务端内部完成校验、幂等、记录和真实退款动作。

模型负责判断用户意图和组织必要参数，系统负责完成确定性流程。

这是边界感。

## 好的 Function Calling，是把不确定性关进笼子里

大模型擅长理解自然语言，也擅长在模糊语境中做判断。但它不擅长承担确定性责任。

退款、下单、转账、删数据、改权限，这些动作都需要可验证、可追踪、可回滚。不能因为模型“觉得应该这样”就直接执行。

所以，Function Calling 的工程原则可以总结成几句话：

工具要少而清楚。不要把内部系统的复杂性原样暴露给模型。

参数要窄而明确。能枚举就枚举，能默认就服务端默认，能从上下文取就不要让模型填。

错误要可读。让模型知道失败原因，才有机会在下一轮修正或解释。

结果要结构化。给用户看的归用户，给系统读的归系统。

高风险动作要由服务端兜底。模型可以建议调用，不能绕过鉴权、校验、幂等和风控。

Function Calling 真正带来的变化，不是让模型替代业务系统，而是让自然语言和业务系统之间多了一层可编排的理解层。

这层理解层做得好，用户说“这个东西根本没法用”，系统就能先确认质量问题，再主动退款，再给出清晰闭环。

做得不好，它就会变成一个会编参数、会误触发、还很难排查的黑盒。

所以我更愿意把 Function Calling 理解为一种协议设计能力：你不是在教模型调用函数，而是在设计模型、应用、工具、用户之间如何可靠地交接责任。
